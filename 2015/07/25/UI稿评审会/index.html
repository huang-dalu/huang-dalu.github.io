UI稿评审会

2015-07-25

记录一下设计同学的 UI 评审会。分享一下。

评审前准备。设计师根据交互自查表对设计稿进行自查。然后准备好相关文档：含筹备阶段的调研整理、竞品分析，以及设计目标、设计方案等。提前一天通过邮件的形式发布评审邀请，预定好会议室。邮件内容包含产品原型、准备好的相关文档、会议室信息。会议时间控制在 1 小时内。

评审中，流程是描述设计目标、展示设计方案参会者讨论设计方案、设计师总结评审结果。演示者负责进行设计方案展示并解释背后思路的人，他同时需要回答所有的问题，或找出能够回答评审提问的人。还有一个角色（也可以是演示者）需要针对当前评审描述项目目标，快速提醒每个人项目目标以及评审涉及的范围。确保整个评审集中在手头任务上，偏离话题。另外，说一个细节，询问设计师如何做出某个方案，应该是“这个方案你是如何想出来的？”而不是“为什么做某个方案？”，需要聚焦在设计思路上。

评审之后就是调整修改和信息同步。

实际工作中，设计师都会对相关竞品以及项目背景进行梳理，但是可能未必有文档记录。这里根据情况判断，不一定非得要求项目背景都整理出来，可以讲解下整体思路即可。

还有项目比较赶的情况。参与评审的设计师手头项目紧急或者产品开发已经同步进行。这时候由产品以及内部几个设计师一起小范围商量讨论，看是否有疑问点以及大的问题等。如果没有，可以不发起评审。这属于特殊情况。

我自己觉得 UI 稿评审是必须有，特别是有 5 人以上的设计师团队。意义在于团队之间取长补短。然后团队设计理念/设计原则/设计规范统一。特别是后者。
